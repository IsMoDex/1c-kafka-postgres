# Принятые архитектурные решения (ADR-стиль)

## ADR-1. Kafka в режиме KRaft (без ZooKeeper)
**Решение:** запускать Kafka 3.8 в KRaft-режиме.
**Причина:** меньше контейнеров и движущихся частей для локального демо; ZooKeeper
устарел. Для теста репликация = 1.
**Компромисс:** не production-grade (одиночный брокер), что и требует ТЗ (раздел 13).

## ADR-2. Собственный consumer-service вместо Kafka Connect
**Решение:** Вариант А ТЗ — собственный сервис.
**Причина:** полный контроль над upsert, идемпотентностью по времени, retry, DLQ,
транзакциями и порядком применения FK-зависимых сущностей (формы → контрагенты).
Kafka Connect JDBC Sink потребовал бы staging-таблицы и SQL-нормализации из-за
конверта события (`event_type`/`payload`), что усложнило бы решение.

## ADR-3. Абстракция источника (Source) + mock
**Решение:** интерфейс `Source` с реализациями `OneCHttpSource` и `MockSource`.
**Причина:** контур Kafka → PostgreSQL должен демонстрироваться независимо от
готовности/доступности 1С (разрешено ТЗ). Реальная 1С — приоритетный источник,
mock — воспроизводимая страховка для демо/CI.
**Компромисс:** дополнительный слой абстракции, но он минимален и оправдан.

## ADR-4. HTTP-сервис 1С как основной способ доступа
**Решение:** собственный HTTP-сервис 1С (Вариант Б ТЗ), OData — как альтернатива.
**Причина:** HTTP-сервис отдаёт чистый JSON, точно совпадающий с payload события;
простой `?changed_since=` для инкремента; не нужно разбирать «толстый» OData-ответ.
**Альтернатива:** OData (Вариант А) — описан в architecture.md, подключается новой
реализацией `Source` без изменения прочего кода.

## ADR-5. Инкремент через реквизит `ДатаИзменения` + watermark
**Решение:** добавить в справочники 1С реквизит `ДатаИзменения` (заполняется в
`ПередЗаписью`), хранить source-based watermark в таблице `sync_state` и читать
с overlap в одну секунду.
**Причина:** справочники 1С не имеют штатного `updated_at`.
**Компромисс:** удаление физической строки в 1С не породит события — поэтому
удаление реализуется как **пометка удаления** (мягкое удаление), что соответствует ТЗ.

## ADR-6. Идемпотентность по времени в upsert
**Решение:** `ON CONFLICT DO UPDATE ... WHERE EXCLUDED.source_updated_at >= stored`.
**Причина:** при повторной/внеочередной доставке не затирать более свежую запись
более старой версией.

Входящий NULL timestamp не имеет права перезаписать существующий непустой
timestamp; consumer требует `updated_at` в Kafka payload.

## ADR-9. Barrier между FK-зависимыми топиками
**Решение:** после flush форм producer ждёт их применения в PostgreSQL и только
затем публикует контрагентов.
**Причина:** Kafka не гарантирует порядок между разными топиками. Barrier даёт
детерминированную гарантию для текущей двухэтапной синхронизации.

## ADR-7. at-least-once + идемпотентный upsert
**Решение:** ручной commit offset после успешной записи; дубли гасятся upsert-ом.
**Причина:** проще и надёжнее exactly-once; при сбое возможна повторная обработка,
но она безопасна благодаря идемпотентности.

## ADR-8. id форм собственности — строковый код
**Решение:** `ownership_forms.id` = `ooo/ip/ao/pao` (TEXT), id контрагентов = GUID (UUID).
**Причина:** соответствие схеме ТЗ (`ownership_forms.id TEXT`, `counterparties.id UUID`)
и удобные стабильные ключи для FK.
