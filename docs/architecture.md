# Архитектура интеграционного контура

## 1. Общая схема

```
┌──────────┐  HTTP/OData   ┌─────────────────────┐  produce  ┌─────────┐  consume  ┌──────────────────┐  upsert  ┌────────────┐
│  1С 8.3+ │ ─────────────▶│ integration-service │ ─────────▶│  Kafka  │ ─────────▶│ consumer-service │ ────────▶│ PostgreSQL │
│ (Windows)│  справочники  │     (producer)      │  события  │ (KRaft) │  события  │   (consumer)     │ ON CONFLICT│           │
└──────────┘               └─────────────────────┘           └─────────┘           └──────────────────┘          └────────────┘
                                    │                                                        │
                              sync_state (watermark)                                    *.dlq (poison)
```

Поток строго соответствует ТЗ: **1С → integration-service → Kafka → consumer-service → PostgreSQL**.

## 2. Компоненты

| Компонент | Роль | Тип запуска |
|---|---|---|
| **1С 8.3+** (реализовано на 8.5.1.1302) | Источник справочных данных (собственный HTTP-сервис) | Вне Docker, на хосте |
| **integration-service** | Producer: читает 1С, публикует события в Kafka | CLI, разовый запуск (`full`/`incremental`) |
| **Kafka (KRaft)** | Шина данных / event streaming layer | Демон в Docker |
| **consumer-service** | Consumer: читает Kafka, upsert в PostgreSQL | Демон в Docker |
| **PostgreSQL** | Целевая интеграционная БД | Демон в Docker |
| **kafka-ui** | Просмотр топиков/сообщений | Демон в Docker |
| **migrate / kafka-init** | Одноразовая инициализация схемы и топиков | one-shot |

## 3. Абстракция источника

`integration-service` работает через интерфейс `Source` (`sources/base.py`):

- `OneCHttpSource` — реальная 1С по HTTP-сервису (приоритет ТЗ);
- `MockSource` — воспроизводимый встроенный источник (демо/CI без запущенной 1С).

Переключение — переменной `SOURCE_TYPE=onec|mock`. Остальной код (producer,
sync, форматы) не зависит от выбора источника. Это гарантирует, что контур
Kafka → PostgreSQL демонстрируется независимо от готовности 1С.

## 4. Модель синхронизации

### Полная (`full`)
Выгружаются все записи справочников без фильтра. Идемпотентность на стороне
PostgreSQL (`ON CONFLICT DO UPDATE`) → повторный запуск не создаёт дублей.

### Инкрементальная (`incremental`)
1. Верхняя граница окна `window_end` фиксируется **до** чтения источника.
2. Из `sync_state` читается `last_synced_at` по каждому справочнику.
3. У источника запрашиваются записи с `changed_since = last_synced_at`.
4. После успешного `flush()` в Kafka watermark продвигается до `window_end`.

Порядок: формы собственности публикуются раньше контрагентов (FK-зависимость).

> В 1С поле «дата изменения» из коробки отсутствует, поэтому в конфигурацию
> добавляется реквизит `ДатаИзменения`, заполняемый в обработчике `ПередЗаписью`.
> Это и есть основа `changed_since`. Ограничения — см. `limitations.md`.

## 5. Идемпотентность и порядок

- **Ключ сообщения Kafka** = `id` объекта 1С → сообщения одного объекта попадают
  в одну партицию, порядок сохраняется.
- **Продюсер**: `enable.idempotence=true`, `acks=all` — без дублей на брокере.
- **Консьюмер**: `INSERT ... ON CONFLICT (id) DO UPDATE` + условие по
  `source_updated_at` (не затираем более свежую запись более старой).
- **Commit offset** — только после успешной записи в PostgreSQL (at-least-once).

## 6. Обработка ошибок и устойчивость

| Сбой | Поведение |
|---|---|
| Недоступна 1С | Ошибка логируется (JSON), CLI завершается кодом 1, watermark не двигается |
| Ошибка доставки в Kafka | Логируется, синхронизация прерывается с ненулевым кодом |
| Временная ошибка PostgreSQL | Retry с экспоненциальной задержкой (до `MAX_RETRIES`) |
| Невалидное сообщение (JSON/схема) | → DLQ (`<topic>.dlq`), offset двигается, поток не блокируется |
| Батч не записался после retry | Весь батч → DLQ, offset двигается |
| FK: контрагент ссылается на несуществующую форму | Retry, затем DLQ (в демо порядок гарантирован) |

## 7. Транзакционность

Батч сообщений записывается в **одной транзакции** PostgreSQL: либо весь батч
применён и offset закоммичен, либо откат и retry. Формы собственности внутри
батча применяются перед контрагентами.

## 8. Почему такие технологии

- **Kafka KRaft** — без ZooKeeper, меньше контейнеров, проще локальный запуск.
- **Python** — быстрая читаемая реализация; `confluent-kafka` (librdkafka),
  `psycopg` v3, `pydantic` v2 для валидации, `structlog` для JSON-логов.
- **Собственный consumer** (а не Kafka Connect) — полный контроль над upsert,
  retry, DLQ, транзакциями и порядком применения FK-зависимых сущностей.
