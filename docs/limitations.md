# Ограничения решения и что улучшил бы в промышленной версии

## Ограничения (осознанные, соответствуют разделу 13 ТЗ)

1. **Одиночный брокер Kafka**, репликация = 1, KRaft одним узлом.
   Не отказоустойчиво — это демо-контур, не production-кластер.
2. **Без авторизации OAuth/JWT.** Доступ к 1С — Basic Auth; Kafka/PG — PLAINTEXT
   в изолированной docker-сети. Секреты только через ENV/.env.
3. **Мягкое удаление вместо реального.** Физически удалённая в 1С запись не
   порождает событие (нечего выгружать). Поэтому удаление = пометка удаления
   (`deleted=true`). Для реального удаления в промышленном варианте нужен план
   обмена 1С или журнал регистрации.
4. **Инкремент опирается на реквизит `ДатаИзменения`.** Если запись меняется в
   обход `ПередЗаписью` (прямой SQL, обмен), watermark может «пропустить» её.
5. **Простая модель данных** — два справочника, без сложных объектов 1С.
6. **Порядок FK** обеспечивается PostgreSQL-барьером: формы публикуются первыми,
   producer ждёт их применения consumer-ом и лишь затем публикует контрагентов.
   Это связывает producer с доступностью целевой БД, но устраняет гонку топиков.
7. **Платформа 1С 8.5.1.1302 Community** вместо приведённой в ТЗ 8.3.
   Совместимость приложенной конфигурации с платформой 8.3 не заявляется. Версия
   8.5 выбрана как доступная среда, где воспроизводимо проверены файловая ИБ,
   конфигурация со справочниками, HTTP-сервис и публикация на веб-сервере.
   Стандартный интерфейс OData в этой сборке требовал ручной установки состава
   объектов (headless-команда `SetStandardODataInterfaceContent` вела себя
   нестабильно), поэтому выбран собственный HTTP-сервис (Вариант Б ТЗ). OData
   остаётся альтернативой, подключаемой отдельной реализацией `Source`.
8. **`ownership_form_id` = стабильный код `ooo/ip/ao/pao`.** Маппинг код↔id зашит
   в модуле HTTP-сервиса 1С. Если в справочник добавить форму собственности с
   кодом вне этого набора, для неё `ownership_form_id` станет GUID объекта 1С
   (fallback). Схема БД (`ownership_forms.id TEXT`) это допускает, но demo/доки
   рассчитаны на 4 предопределённых кода.
9. **`POST /seed` сбрасывает только известные demo-записи.** Он восстанавливает
   поля четырёх форм и пяти контрагентов и снимает с них пометку удаления, но не
   удаляет дополнительные записи, вручную созданные в 1С. Mock-состояние отдельно
   хранится в файле `SEED_STATE_PATH`.
10. **Сеть Docker → 1С.** Основной адрес — `host.docker.internal`. В одном
    состоянии Docker Desktop/IIS он отдавал 502 на ISAPI-ответ 1С; fallback —
    реальный IPv4 хоста. IP может меняться при перезагрузке — тогда обновите
     `ONEC_BASE_URL`.
11. **Timestamp-инкремент.** Для защиты секундной точности используется overlap
    в одну секунду. Повторно прочитанные записи сравниваются с PostgreSQL и не
    публикуются в Kafka, если состояние не изменилось. При ручном переводе часов
    1С назад нужен сброс watermark или full sync; промышленный вариант — план
    обмена.
12. **Offset pagination 1С.** HTTP-сервис использует стабильную сортировку и
    `limit/offset`, но изменение набора между страницами не даёт snapshot isolation.
    Overlap и идемпотентность снижают риск для incremental; для больших нагрузок
    нужен cursor `(ДатаИзменения, Ссылка)` или snapshot token на стороне 1С.
13. **Compose — production-like, не production deployment.** Основной файл
    запускает non-root `prod`-образы и Flyway, но Kafka остаётся одноузловой и
    использует PLAINTEXT. Test stages подключаются только через `compose.test.yaml`.

## Что улучшил бы в промышленной версии

- **Schema Registry (Avro/Protobuf)** вместо «сырого» JSON — контроль эволюции схемы.
- **План обмена 1С** для честной регистрации изменений и удалений (включая
  физические), вместо реквизита `ДатаИзменения`.
- **Exactly-once**: транзакционный producer + `read_committed`, либо outbox-паттерн
  на стороне интеграции.
- **Kafka-кластер** (3+ брокера, replication factor 3, min.insync.replicas=2).
- **Безопасность**: TLS + SASL/SCRAM для Kafka, TLS для PostgreSQL, секреты в
  Vault/Secret Manager, OAuth для HTTP-сервиса 1С.
- **Наблюдаемость**: `/metrics` уже отдаёт readiness, throughput, retries и DLQ;
  промышленному контуру нужны consumer lag по partition, histogram latency,
  OpenTelemetry и алертинг.
- **Мониторинг DLQ** и автоматизированный re-drive (переобработка).
- **Миграции** выполняются Flyway с историей и checksum; для промышленной среды
  нужно отключить переходный `baselineOnMigrate` после обновления старых volumes.
- **CDC**: при больших объёмах — Debezium от журнала 1С/СУБД вместо poll-синхронизации.
- **Backfill/replay**: возможность перечитать топики с нуля для восстановления БД.
- **Тесты**: есть реальные PostgreSQL integration tests; дополнительно нужны
  нагрузочные тесты и mutation/snapshot тест keyset-пагинации 1С.
